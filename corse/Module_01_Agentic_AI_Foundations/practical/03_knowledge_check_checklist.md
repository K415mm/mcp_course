## Section B: Operational Pre-Deployment Checklist

Before deploying any MCP-enabled AI agent in a SOC environment, complete this checklist. Each item must be signed off before the agent goes live.

### Agent Scope
- [ ] The agent's goal is written in a single sentence. (If you can't do this, the scope is too broad.)
- [ ] Every tool the agent has access to has been classified as Sense-only or Act.
- [ ] Every Act tool has a defined approval gate or is forbidden entirely.
- [ ] Every Act tool has a documented rollback procedure.

### Cynefin Classification
- [ ] All alert types the agent will handle have been Cynefin-classified.
- [ ] Act permissions exist only for alerts classified as Clear or (with approval gate) Complicated.
- [ ] Complex and Chaotic alert types have a documented "agent pauses, human decides" protocol.
- [ ] A procedure exists for what happens if the Cynefin domain cannot be determined (Confusion): agent is deactivated, human classifies first.

### Tool Safety
- [ ] All tools have type hints and descriptive docstrings.
- [ ] All tools return `{"status": "ok"}` on success and `{"status": "error", "reason": "..."}` on failure.
- [ ] All tools validate their inputs before executing.
- [ ] All destructive tool docstrings include `[DESTRUCTIVE] REQUIRES HUMAN APPROVAL`.
- [ ] Output normalization applied (no "suspicious/malicious/dangerous" in returned text).

### Human-in-the-Loop
- [ ] Approval gate is a formal structured request, not a natural language message.
- [ ] Approval requires an explicit identifier (analyst name/ID) in addition to "yes/approved."
- [ ] Approval is logged to the SIEM within 5 seconds of the interaction.
- [ ] An "off-switch" exists and is documented: any authorized analyst can halt the agent within 60 seconds.

### Audit and Monitoring
- [ ] Every tool call is logged: timestamp, tool name, input parameters, output status, caller.
- [ ] The audit log is ingested into the SIEM.
- [ ] A dashboard exists to review agent activity (tool call counts, approval rates, error rates).
- [ ] A review schedule is defined: first week daily, then monthly.

## Module 01 Knowledge Check: Answer Key (Instructor)

**Q1:** A tool call is a structured JSON request generated by the LLM naming a specific function and its arguments. The framework (not the LLM) executes the function. The LLM only produces text that describes the call; it does not execute code directly.

**Q2:** "The agent's enrichment tools gathered data about the IP (Sense). The LLM processed the gathered context and assessed that the probability distribution of outcomes pointed toward a block action (Think). The framework called the `block_ip` tool (Act)."

**Q5 examples:** (1) A novel attack pattern that has no training data analogue — the LLM produces a plausible-looking analysis that is statistically reasonable but empirically wrong. (2) Amplification of emotionally-loaded tool outputs — an AI will read "suspicious API call" and increase its risk verdict even when the API is commonly used by legitimate software.

**Q9:** Requires knowing: (a) What Cynefin domain is associated with the alert type triggering isolation? (b) Are those 3 corroborating signals independently sourced (3 sources) or from the same data source (less reliable)? (c) Is the endpoint production-critical? The proposal is plausible for Clear-domain alerts — risky for Complicated, unacceptable for Complex.
